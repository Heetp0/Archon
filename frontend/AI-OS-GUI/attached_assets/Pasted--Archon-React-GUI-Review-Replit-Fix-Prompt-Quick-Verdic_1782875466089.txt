# Archon React GUI — Review & Replit Fix Prompt

## Quick Verdict

This is genuinely a strong foundation — better than either the CustomTkinter
or the mid-build PyQt6 attempt. Real WebSocket reconnect logic with
exponential backoff, a clean typed message protocol, resizable panels,
full shadcn/ui component library, and actual state management via Context.
Worth committing to.

There is **one critical mismatch** and a handful of the same "mock data
presented as real" issues from the other GUI attempts. All fixable.

---

Anthropic, and Google's consumer APIs — not the free-tier stack Archon
actually uses.

## Other Issues Found

### 1. Duplicate/dead sidebar component
`LeftSidebar.tsx` exists and duplicates the functionality of `NavRail.tsx`
+ `ContextSidebar.tsx` combined into one file. `Home.tsx` only imports
`NavRail` and `ContextSidebar` — `LeftSidebar.tsx` is never used anywhere.
Dead code, should be deleted.

### 2. Mock data mixed with real data (same pattern as the CTk version)
- `ChatMode.tsx` — `MOCK_MESSAGES` array is concatenated with real
  `chatMessages` from the WebSocket on every render: `[...MOCK_MESSAGES, ...chatMessages]`.
  This means fake conversation history is permanently shown above real messages.
- `ContextSidebar.tsx` / `LeftSidebar.tsx` — `MOCK_FILES` is a hardcoded
  5-file list, never connected to the real vault.
- `AppContext.tsx` — `MOCK_HISTORY` is 6 hardcoded fake session entries.
- `WebSocketContext.tsx` — `agentStatuses` and `taskQueue` initialize with
  hardcoded fake running agents (`Security_Scanner_01`, task IDs `T-019`
  through `T-023`) before any real daemon connection.
- `WebSocketContext.tsx` — `researchText` initializes with a hardcoded
  fake analysis paragraph.
- `ResearchMode.tsx` — the entire knowledge graph (`NODES`, `EDGES`) is
  a hardcoded fake graph about "Quantum Error Correction" with fictional
  node positions. This isn't a placeholder — it's fake finished content
  masquerading as a real result.

### 3. Message protocol mismatch with the actual daemon
The frontend's `useWebSocket.ts` expects message types like `chat_chunk`,
`chat_complete`, `agent_status`, `council_chunk` — but the real Archon
daemon (per `daemon/main.py` and `daemon/council_debate.py`) uses a
different event envelope structure (`{id, event, payload}` with event
types like `token`, `status`, `gate`, `done`, `error`). These two protocols
will not talk to each other without a translation layer on one side or
the other.

### 4. Settings don't persist anywhere
`SettingsModal.tsx` has local `useState` for the API key input but no
save mechanism — no `localStorage`, no POST to a daemon endpoint. Typing
a key and closing the modal loses it immediately.

### 5. Port/host settings are decorative
The Network settings tab has inputs for port (`8765`), host (`localhost`),
and timeout (`2000`) with `defaultValue` — these are uncontrolled inputs
that don't feed back into `useWebSocket.ts`'s hardcoded
`const DAEMON_URL = "ws://localhost:8765"`. Changing them does nothing.

---

## Design Observations (Good, Not Bugs)

- The color-coded glow system per mode (blue/orange/purple/green) is a
  nice touch and more polished than either Python GUI attempt
- Resizable right sidebar with drag handle is a genuinely useful feature
  neither other version has
- Connection status indicator in the top bar (`Wifi`/`WifiOff`/spinner)
  is exactly the kind of real-time daemon-state feedback that was missing
  elsewhere
- Terminal mode's line-kind coloring (system/input/output/warning/error/success)
  is a nice implementation

## FIX 4 — Fix message protocol mismatch with the real daemon

The daemon uses this envelope format for all WebSocket messages:
```json
{ "id": "...", "event": "token" | "status" | "gate" | "done" | "error", "payload": {...} }
```

The frontend currently expects flat message types like chat_chunk,
chat_complete, agent_status, council_chunk directly on the top-level
message object (see src/hooks/useWebSocket.ts WSMessage union type and
src/context/WebSocketContext.tsx's switch statement).

### File: src/hooks/useWebSocket.ts

Update the WSMessage type to match the daemon's actual envelope:
```tsx
export interface DaemonEnvelope {
  id: string;
  event: "token" | "status" | "gate" | "done" | "error";
  payload: Record<string, unknown>;
}
```

In the onmessage handler, parse the envelope and pass it through as-is —
do not try to flatten it here. Let WebSocketContext.tsx handle interpreting
the event/payload structure.

### File: src/context/WebSocketContext.tsx

Rewrite the switch statement in the useEffect that processes ws.messages.
Instead of switching on m.type with flat fields, switch on m.event and
read from m.payload. The exact payload shape depends on which daemon
route sent it — coordinate with the daemon's send_token_callback and
WebSocket handler in daemon/main.py to confirm exact payload field names
for each event type (agent name for council tokens, model name for chat,
etc). Add a fallback case that logs unrecognized events to the console
rather than silently dropping them, to make future debugging easier:
```tsx
default:
  console.warn("Unhandled WebSocket event:", m.event, m.payload);
```

## FIX 5 — Persist Settings to localStorage (interim solution)

### File: src/components/SettingsModal.tsx

API keys currently exist only in local component state and vanish when
the modal closes. Add localStorage persistence as an interim solution
until a proper backend settings endpoint exists.

In the KeyRow component, replace:
```tsx
const [key, setKey] = useState("");
```
with:
```tsx
const storageKey = `archon_apikey_${provider.value}`;
const [key, setKey] = useState(() => localStorage.getItem(storageKey) || "");

useEffect(() => {
  if (key) {
    localStorage.setItem(storageKey, key);
  } else {
    localStorage.removeItem(storageKey);
  }
}, [key, storageKey]);
```
Add `useEffect` to the React import at the top of the file if not already
present.

Note: this is a temporary client-side solution. Keys stored this way are
only available in the browser session and won't be visible to the Python
daemon. A follow-up task (not part of this fix) will be to add a
POST /settings/api-keys endpoint on the daemon and wire this form to call
it via fetch instead of localStorage.
